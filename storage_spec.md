# Спецификация: Интеграция Supabase Storage

## 1. Обзор
Переход от локального хранения медиа-файлов (скриншоты, видео) к облачному хранилищу Supabase Storage для обеспечения доступности данных пользователям и масштабируемости системы.

## 2. Инфраструктура Storage
- **Бакеты (Buckets)**:
    - `screenshots`: Публичный, для хранения PNG-файлов (скриншоты ошибок).
    - `videos`: Публичный, для хранения MP4-файлов (записи тестов).
- **Политики доступа (RLS)**:
    - **Service Role**: Полный доступ (INSERT, SELECT, DELETE) для воркера бэкенда.
    - **Public**: Доступ только на чтение (SELECT) для отображения медиа на фронтенде.

## 3. Логика загрузки (Infrastructure Layer)
- **Библиотека**: `supabase-py` (использование асинхронного клиента).
- **Метод**: `storage.from_(bucket).upload(path, file)`.
- **Content-Type**: Автоматическое определение (image/png, video/mp4).
- **Шаблон именования**: `scan_{scan_id}/{timestamp}_{random_hash}.{ext}`.
- **Публичные ссылки**: Использование `get_public_url()` после загрузки.

## 4. Связь с базой данных (Service Layer)
- После загрузки Public URL записывается в таблицу `test_results` (поле `screenshot_url` или `video_url`).
- Связь осуществляется через `test_id` (или `scan_id`).

## 5. Жизненный цикл файлов
- **Временные файлы**: Локальные файлы в `/tmp` (или `screenshots/`, `videos/`) удаляются сразу после подтверждения успешной загрузки в Storage.
- **Ошибки**: Если загрузка не удалась, в лог БД записывается ошибка, но процесс тестирования продолжается.

## 6. Безопасность
- Использование `SUPABASE_SERVICE_ROLE_KEY` только на бэкенде.
- Отсутствие прямых ключей в коде (использование `.env`).
